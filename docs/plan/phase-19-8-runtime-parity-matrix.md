## Step 1 — Driver scaffolding (Deno)

**Goal.** Two passing Deno-driven parity files: one against the Node
adapter (via dist/esm/index.node.js) and one against the Memory adapter
(via dist/esm/index.default.js).

Files:

1. `test/runtime-parity/deno/deno.json`
   - `compilerOptions.allowImportingTsExtensions` ON.
   - Minimal — no remote imports, no `import_map.json` (drivers use only
     relative paths + the Node-compat ambient surface).

2. `test/runtime-parity/deno/parity-memory.test.ts`
   - Import `SCENARIOS` from `../../parity/scenarios/index.ts`.
   - Import `openRepository` from `../../../dist/esm/index.default.js`.
   - For each scenario, call `Deno.test({ name, fn: async () => {...} })`.
   - Stage inputs into the Memory adapter via
     `openRepository({ files: stageFiles(scenario.inputs) })`, mirroring
     `test/parity/memory.test.ts`.
   - Use `assertEquals` from `jsr:@std/assert` (Deno-blessed; resolved at
     test time without an import map).

3. `test/runtime-parity/deno/parity-node.test.ts`
   - Same shape as `parity-memory.test.ts` but staging real fs via
     `node:fs/promises` (Deno's Node-compat surface) and importing
     `openRepository` from `dist/esm/index.node.js`.

**Verify locally.** `deno test --allow-read --allow-write --allow-env
--no-prompt test/runtime-parity/deno/` — every scenario passes against
both adapters.

**Commit.** `feat(parity): deno runtime drivers`.

## Step 5 — CI matrix jobs

File modified: `.github/workflows/ci.yml`.

Add three jobs immediately after `parity-tests`:

```yaml
parity-deno:
  needs: [changes, build]
  if: needs.changes.outputs.code == 'true'
  runs-on: ubuntu-latest
  steps:
    - uses: actions/checkout@v6
    - uses: denoland/setup-deno@v2
      with: { deno-version: v2.x }
    - uses: actions/download-artifact@v7
      with: { name: dist, path: dist/ }
    - run: deno test --allow-read --allow-write --allow-env --no-prompt test/runtime-parity/deno/

parity-bun:
  needs: [changes, build]
  if: needs.changes.outputs.code == 'true'
  runs-on: ubuntu-latest
  steps:
    - uses: actions/checkout@v6
    - uses: oven-sh/setup-bun@v2
    - uses: actions/download-artifact@v7
      with: { name: dist, path: dist/ }
    - run: bun test test/runtime-parity/bun/

parity-workers:
  needs: [changes, build]
  if: needs.changes.outputs.code == 'true'
  runs-on: ubuntu-latest
  steps:
    - uses: actions/checkout@v6
    - uses: ./.github/actions/setup
    - uses: actions/download-artifact@v7
      with: { name: dist, path: dist/ }
    - run: npx vitest run --config test/runtime-parity/workers/vitest.config.ts
```

All three are blocking (no `continue-on-error`).

**Verify.** Push branch, observe matrix on the PR.

**Commit.** `ci: parity-deno / parity-bun / parity-workers matrix`.

## Sequencing notes

- Steps 1, 2, 3 are independent — Deno, Bun, and Workers drivers do not
  depend on each other. They can be implemented in parallel by separate
  agents if the sub-worktree strategy is invoked, but the integration
  surface is small enough that serial implementation is faster.
- Step 4 (devDependencies) depends only on Step 3 finishing
  (Cloudflare's pool is the only new dep).
- Step 5 (CI) depends on Steps 1–4.
- Steps 6–8 finalise and can run after Step 5 is green in CI.
