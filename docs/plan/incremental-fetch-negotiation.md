## Part 1 — v1 fallback framing fix + un-strip `multi_ack_detailed`

### Context

Realizes ADR-451 (v1 fallback correctness) + design §4. Smallest, self-contained,
independently valuable: delivers incremental fetch on the v1 path immediately (wire matrix
row A→B). Depends on nothing; Part 3 later reuses the new `encodePktLines`.

**Root cause (pinned, design §"Empirical root cause"):** `buildUploadPackRequest` builds the
have-list with `encodePktStream(...)`, which **unconditionally appends a `0000` flush**
(`src/domain/protocol/pkt-line.ts:68`, `out.set(FLUSH_PKT, off)`). For `haves:[…]` +
`done:true` the wire becomes `want … / 0000 / have … / 0000 / done` — the server reads
`have C0 / 0000` as a non-final round (`compute-end = flush-pkt`), answers ACKs and **no
pack**, ignores the trailing `done`. Correct grammar: `compute-end = flush-pkt / "done"` —
EITHER a flush OR done, never both.

**Files/symbols to touch:**
- `src/domain/protocol/pkt-line.ts` — ADD `export const encodePktLines = (payloads: ReadonlyArray<Uint8Array>): Uint8Array`:
  concat each payload via the same length-prefix logic as `encodePktStream` **but WITHOUT
  the terminating `FLUSH_PKT`**. Mirror the existing `MAX_PKT_LINE_PAYLOAD` per-payload
  `RangeError` guard. (Public — barrel it alongside the `encodePktLine`/`encodePktStream`
  family; see gate below.) Existing helpers: `encodePktLine` (l.36), `encodePktStream`
  (l.49), `lengthPrefix` (l.30), `FLUSH_PKT` (l.26), `PKT_LENGTH_BYTES` (l.23).
- `src/domain/protocol/index.ts` — add `encodePktLines` to the `./pkt-line.js` export block
  (currently exports `DELIM_PKT, decodePktStream, encodePktLine, encodePktStream, FLUSH_PKT,
  GitExchange, MAX_PKT_LINE_PAYLOAD, PktLine, RESPONSE_END_PKT`).
- `src/domain/protocol/upload-pack.ts` — `buildUploadPackRequest` (l.283). Change the
  have-list terminator to depend on `done`:
  - `haves` empty → unchanged (`haveStream = new Uint8Array(0)`; clone = `want-list flush [done]`).
  - `done === true` **and** `haves` non-empty → `haveStream = encodePktLines(haves.map(haveLine))`
    (no trailing flush); the existing `DONE_FRAME` (l.281) follows. Wire:
    `want-list flush have-list done`.
  - `done` falsy → keep `encodePktStream(...)` (have-list terminated by a flush — genuine
    negotiation round). `wantStream`/`trailer`/concatenation (l.291-302) are otherwise
    unchanged. `haveLine` (l.275) unchanged.
- `src/application/commands/internal/upload-pack-client.ts` — `selectFetchCapabilities`
  (l.37). DROP `multi_ack_detailed` from the filter predicate (l.41), keeping `thin-pack`,
  `no-progress`, and `AGENT` filtered. The parser (`parseAckLine`, upload-pack.ts l.311)
  already tolerates the resulting `ACK … common` lines. Update the doc comment (l.30-36,
  l.7-11) that claims "no multi_ack_detailed" so it stops lying. Single-round strategy is
  retained (all ≤256 haves + `done` in one POST). `CLIENT_CAPABILITIES_FETCH`
  (`capabilities.ts:6`) already lists `multi_ack_detailed`.

**Seam:** pure domain request-builder + one client capability-filter tweak. No I/O.

**Surface gates:** `encodePktLines` is a NEW public export on the protocol barrel →
regenerate + commit `reports/api.json` in this part.

## Part 3 — v2 pkt framing + section decoder + capability-advertisement parser

### Context

  - `export const encodeCommandRequest = (command: string, args: ReadonlyArray<string>,
    payloads: ReadonlyArray<Uint8Array>): Uint8Array` — emits the v2 request framing:
    command-header pkt-lines (`command=<command>\n`, `agent=<AGENT-value>\n`,

    section dispatcher over the **v2-flagged** decode stream: reads a data pkt-line as a

    caller-supplied handler set. Recognises `{acknowledgments, shallow-info, wanted-refs,
    packfile}`; an unrecognised section header throws `unexpectedV2Section`. (Parts 4/5 supply

## Risks for the implementers

- **api.json cadence (Parts 1–6).** typedoc resolves `src/domain/index.ts` (which `export *`s
  the protocol barrel) and `src/application/primitives/index.ts`, so EVERY part that adds a
  protocol/primitives export changes `reports/api.json`. `validate` stays green while the
  `prepush` `check:doc-typedoc` gate rejects — regenerate (`npm run docs:json`) and commit
  api.json IN each part (the gate above runs `check:doc-typedoc` to catch it in-part).
- **Exhaustiveness switch (Part 3).** The two new `ProtocolError` codes MUST get `case` arms
  in `extractDetail` (`src/domain/error.ts`) or `check:types` fails on the `never` default —
  and each arm must be executed by a test (`src/domain/error.ts` is coverage-gated).
- **Response-driven version detection (Part 6).** The peek-and-pushback on the advertisement's
  first data pkt-line (reusing the `pushbackBuffer` WeakMap in `upload-pack.ts`) is the load-
  bearing fallback mechanism (R7). Get the prologue-consume-then-peek order right, mirroring
  `consumeServiceHeader`; a mis-consumed prologue corrupts both legs.
- **Empty pack still verifies (Part 2).** A 0-object pack is a 32-byte header+trailer;
  `verifyPackTrailer` must pass BEFORE the `entries.length === 0` guard — order the guard after
  verify/walk, not before. Keep the pre-existing zero-byte guard (distinct reachable state).
- **`everything_local` partial-repo gate (Part 2).** The short-circuit MUST be gated
  `filter === undefined`; a partial repo's wants can be promised-absent, so short-circuiting
  would skip a needed fetch. Isolate that guard in its own test.
- **Interop flakiness (Part 7).** Scrub `GIT_*` (husky pollutes `GIT_DIR`), sign OFF, pin the
  peer's `protocol.version` and `merge.conflictStyle`, one shared `beforeAll` repo + 60s
  timeout, skip under `.stryker-tmp`. SSH push/fetch fail on `publickey` in the sandbox — the
  SSH leg (P3/optional) may only be runnable in CI.
- **Compression watchdog.** Session output-compression can stall craft subagents; part
  implementers should read compressed tool output via `headroom_retrieve` (hash=) rather than
