# ADR-121: `@proves` header grammar for integration tests

## Status

Accepted (at `9b109c1fecccf317fc4b017127fe6bedf849b26c`)

## Context

Phase 19.4 makes every integration test file declare what it proves so duplicates, orphans, and misplaced files surface as findings. The artifact carrying that claim has to live somewhere the audit can read and the human reviewer cannot miss.

Three placements were considered:

1. **JSDoc header block on the test file itself.** Travels with the test in code review; existing files already open with `/**` so the conversion is amendment, not creation. Parser is a string scan.
2. **Sidecar `.proves.json`** next to each test. Machine-friendly, schema-validatable. Cost: every test file becomes a pair; reviewers must open two files; "tests that don't have a sidecar" become an extra class of finding.
3. **Central manifest** (`integration-surfaces.json`) listing every file. Single source of truth. Cost: every PR that adds an integration test must also touch the manifest; merge conflicts on the manifest become routine; the claim is far from the test it justifies.

## Decision

Use a JSDoc header block with a single `@proves` directive and three required keys: `surface`, `bucket`, `unique`. The block must sit in the first JSDoc comment of the file (optionally after a shebang). Surrounding prose is free-form; only the structured lines are parsed.

Grammar:

- `surface` matches `^[a-z][a-zA-Z0-9.-]{1,40}$`.
- `bucket` is one of the enum values in [ADR-122](122-integration-bucket-taxonomy.md).
- `unique` is single-line text, 12–200 chars.

The audit emits `reports/integration-surfaces.json` as a *derived* artefact (the manifest from placement #3 above) so 19.5a still has a machine-readable index — but the index is generated, never hand-edited.

## Consequences

### Positive

- **Claim travels with the test.** A reviewer reading `clone-http-backend.test.ts` sees the surface and bucket without leaving the file.
- **Zero new files in the sweep PR.** Every existing integration file already has a `/**` opening; the sweep only appends three lines per file.
- **Derived index stays in sync.** Because the JSON is regenerated by the audit, it cannot drift from the headers; a "fix the index" PR is impossible.

### Negative

- **Pure-string parser.** No AST, so the grammar is intentionally narrow (one block, one `@proves`, three keys). Authors who want richer structure will have to wait for a future revision.
- **CRLF / mixed-encoding files would trip the parser** if not normalised. Mitigation: parser normalises line endings on entry (`\r\n` → `\n`) before tokenising.

### Neutral

- **`@proves` is a custom tag, not a TypeDoc-recognised one.** TypeDoc ignores unknown tags by default; the JSDoc block continues to render unchanged in any IDE that displays the comment. No tooling collisions.
