# 486 — Validate the status:clean win with a same-host bench and refresh the full profile baseline

- **Status:** accepted
- **Date:** 2026-07-13
- **Design:** docs/design/status-clean-perf-investigation.md · **Supersedes/Refines:** none

## Context

The containment amortisation ([ADR-485](485-status-clean-containment-tax-amortisation.md))
must be shown to actually move `status:clean`, and the change touches frames
(`resolveForMode` / `checkContainment`) that appear in the committed CPU-profile baseline
(`docs/perf/baseline.json` / `.md`). Two constraints shape how the win is pinned:

- **Local absolute numbers are not citable.** A personal host under interactive load biases
  tsgit's `lstat`-heavy paths (isomorphic-git itself measures 1.2–2.4× slower under load);
  the published `status:clean` number is sourced from the CI nightly runner (ADR-483). But a
  **same-host before/after** ratio *is* load-independent — both sides pay the same
  contention — which is exactly the method that proved "no regression" in ADR-485.
- **The profile baseline is a whole-command artifact.** `npm run profile status` regenerates
  the baseline with **only** the `status` command's section, deleting every other command's
  entry; committing it would silently gut the baseline. A faithful refresh requires
  `npm run profile` over **all** commands. There is currently **no baseline-drift CI gate**,
  so the baseline is a documentation artifact, not a hard gate.

## Options considered

1. **(a)** Same-host before/after `status` bench in the PR body (host-relative, uncommitted);
   do not touch the committed baseline. Pros: load-independent proof, no unsafe artifact. /
   Cons: the committed baseline's `status` self-shares go stale relative to the optimised code.
2. **(b)** Option (a) **plus** a full `npm run profile` (all commands) refresh of the
   committed baseline. Pros: the baseline reflects the optimised `resolveForMode` /
   `checkContainment` frame; the diagnosis and the artifact stay in sync. / Cons: a full
   profile run is expensive.
3. **(c)** CI-nightly only — rely on the next `bench.yml` run to move the published number,
   commit nothing locally. Pros: no manual step. / Cons: cannot attribute the move to this
   change versus runner noise, and the baseline stays stale until the nightly.

## Decision

**Option (b).** The perf win is validated by a **same-host before/after `status` bench**
(branch base vs branch tip, one machine), with the host-relative numbers recorded in the
**PR body** — never committed to the tree. **And** the committed profile baseline is
refreshed with a **full `npm run profile`** (all commands) so it reflects the optimised
containment frame. A **status-only** `npm run profile status` baseline is **never**
committed (it would delete every other command's section). The published `status:clean`
number in `performance.md` stays CI-nightly-sourced (ADR-483) and moves only via a nightly
refresh.

## Consequences

- The PR body carries the load-independent before/after ratio as the win's evidence; the
  reader can reproduce the *ratio* on any host even though the absolute numbers are
  host-biased.
- `docs/perf/baseline.json` / `.md` are regenerated by a full `npm run profile` as part of
  the change and committed; the diff is the expected self-share shift on the optimised
  frames (and run-to-run noise on the rest), not a whole-file rewrite.
- The commit that refreshes the baseline must be a full-profile run — a status-only baseline
  is a known-unsafe artifact and is called out in the design's Out-of-scope so a future
  contributor does not commit one.
- No CI gate is added; the baseline stays a documentation artifact until the 26.5 regression
  gate (out of scope here) formalises a per-scenario `bench:summary` diff.
