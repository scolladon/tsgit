## Phase 21 — High-usage porcelain (v2)

Composition on Phase 20.

- [x] **21.1** `pull` — `fetch` + `merge` on `repo.*`. Strict upstream resolution (`opts.remote ?? branch.<cur>.remote ?? origin`; `opts.branch ?? branch.<cur>.merge`), and `clone` now writes `[remote "origin"]` + `[branch "<head>"]` tracking config for every clone (ADR-196). pull resolves the tracking ref → OID and delegates to `merge` with a faithful `Merge branch '<x>' of <url>` message + a new `merge` `reflogLabel` so the reflog reads `pull:` exactly like git (ADR-197). `merge.resolveTarget` broadened to gitrevisions ref-DWIM (`origin/main`, tags) via a shared `refCandidates` ladder + tag peeling (ADR-199). `rebase` mode omitted until 22.3 (ADR-198). Conflicts compose with the 20.4 state machine (abort/continue) unchanged. Also fixed a latent fetch bug: the per-`Context` pack registry was not refreshed after a pack write. · ADRs 196–199 · `design/phase-21-1-pull.md`
- [x] **21.2** `mv` — atomic rename in index + working tree on `repo.*`. Validate-all-then-execute so a refusal moves nothing; the working file is renamed as-is and the source's index entry (blob id + mode) is copied to the destination (no re-hash, so unstaged edits travel with the file — verified byte-identical to `git mv`). Directory sources reparent every tracked entry and move the subtree leaf-by-leaf (adapter-portable, since memory/OPFS `rename` is file-only). Pinned byte-for-byte by `add-add-content-interop` (12 twin git/tsgit scenarios: per-region markers, empty-ours region, mode-only, binary, union-clean, symlink/symlink, distinct-types both orders, slashed-label flattening, tracked `_0` collision, refusal parity, cherry-pick suffix). · ADRs 310–311 · `design/add-add-content-merge.md` _(surfaced by 24.9a.)_
- [x] **24.9g** Subsection-name (un)escaping in section headers — `parseSectionHeader` now mirrors git's `get_extended_base_var`: `\c` → `c` decoded verbatim inside `[section "subsection"]` (no named escapes — `\t` is `t`), `]`/`#`/`;`/CR literal inside the quotes, and a **strict quoted grammar** (whitespace required before the quote, closing quote immediately followed by `]`) whose violations refuse with `CONFIG_PARSE_ERROR { line, source, partialSectionName }` exactly where git dies — scoped to quoted headers only, unquoted-header malformations stay lenient (ADR-312). The **writer** escapes `\`/`"` (only) via `write_section`'s grammar and relaxes subsection acceptance to git's charset (`"`/`\`/`]`/CR representable; LF/NUL refused), in both `renderSectionHeader` and the dotted-key parser. **Write paths mirror git's per-operation refusal map** (ADR-313): set/unset parse first and translate header malformations to `CONFIG_INVALID_FILE { sectionName, source }` carrying git's partially-accumulated `invalid section name` diagnostic, while value malformations keep the read shape; rename/remove-section drop the full parse and match line-surgically — lenient on malformed files like git's `copy_or_rename`, repairing the ADR-308-era bad-value divergence there. Pinned by `config-interop` (subsection write-parity matrix byte-identical vs git 2.54, exotic-header read parity, same-line read refusals, write-refusal partial-name parity, rename leniency + `no such section` parity) + full-unicode subsection round-trip properties. · ADRs 312–313 · `design/config-subsection-escaping.md` _(surfaced by 24.9c.)_
- [x] **24.9h** Valueless keys — a `key` line with no `=` is now tokenized as a present-but-valueless entry, `value: string | null` with `null` mirroring git's internal NULL (ADR-314): distinct from absent (`undefined`) and empty (`''`), JSON-survivable, rippled through every structured read surface (`IniSection`, `collectValues`, `configGet`/`getAll`/`list`/`getRegexp`, `previousValue`). Boolean reads map `null → true` (`[core]` + `bare` ⇒ bare; `sparsecheckout`, `logallrefupdates`, `promisor`, `submodule.active` inherit via `parseGitBoolean`); string-typed `ParsedConfig` fields treat valueless as absent (ADR-315 — documented divergence from git's lazy `missing value` die, see 24.9l). The no-`=` path now enforces git's `get_value` key grammar (alpha-first alnum-dash, space/TAB-then-EOL, CRLF folded): `key ; c` / `bad!key` / `9key` refuse with `CONFIG_PARSE_ERROR` at git's exact line; `[`-prefixed non-header lines and orphan sectionless keys keep the lenient skip (24.9m). Writer surgery (`isKeyLine`) replaces/removes valueless physical lines (set → canonical `key = value`, pinned bytes); `getRegexp` value-patterns match NULL as `''`. Pinned by `config-interop` (list reconstruction, `rev-parse --is-bare-repository` co-truth, 5-line refusal-parity matrix, set/unset/rename write bytes, get-regexp parity) + grammar-totality/negative/writer-preservation properties. · ADRs 314–315 · `design/config-valueless-keys.md` _(surfaced by 24.9c.)_
- [x] **24.9i** Line-surgery writer vs multi-line entries — rewriting/removing/appending around an existing entry whose value spans backslash continuations operated on the first physical line only (replace left an orphan tail line; append-after-last inserted before the tail). tsgit never writes such entries itself (LF is escaped to `\n`), so this only bit on files written by other tools; fixed by making the writer span-aware via the shared `tokenizeConfig` tokenizer. · ADRs 316–317 · `design/config-multiline-line-surgery.md` _(surfaced by 24.9c.)_
- [x] **24.9j** Distinct types **with a base** — 24.9f covers the no-base (add/add) collision; when a base exists and the two sides diverge to different kinds, tsgit still emits the take-ours `type-change` conflict, while git's ort renames per side like the add/add case. Same shape as 24.9f: per-side recorded paths + symlink-aware writes (today a type-change whose ours side is a symlink writes the link-target bytes as a regular file). _(surfaced by 24.9f.)_ · ADRs 318–321 · `design/distinct-types-with-base.md`
- [ ] **24.9k** Empty subsection `[s ""]` vs `[s]` matching — git treats an explicitly empty subsection as a distinct section (`s.k` does not match `[s ""]`, pinned against git 2.54); tsgit's `matchesSection` conflates them (`subsection === undefined` matches both `[s]` and `[s ""]`), and 24.9i's token-based `matchesTarget` reproduces the conflation verbatim — fix BOTH matchers together (line-based section ops + token-based entry surgery share the identity rule) + pin with interop (set/get/unset against a file holding both forms). _(surfaced by 24.9g; twin matcher noted by 24.9i.)_
- [ ] **24.9l** Per-use-site `missing value` refusal parity — git dies lazily when a *string-typed* key is read with a NULL value (`error: missing value for 'user.name'` + `fatal: bad config variable … at line N`) while `--list` succeeds; ADR-315 scoped 24.9h to treat valueless string fields as absent, so e.g. `commit` refuses via tsgit's identity-not-configured path instead of git's exact message. Thread a missing-value marker (or re-read the raw entries) at the consuming commands and match git's refusal shapes; also covers git's int-typed `bad numeric config value '' … invalid unit` once an int config key is merged. _(surfaced by 24.9h / ADR-315.)_
- [ ] **24.9m** Char-wise config parser parity — the line-wise tokenizer cannot represent constructs git's char-wise parser accepts on one line: `[a] key = v` / `[a] key` (header **plus** entry on the same physical line — git records `a.key`; tsgit lenient-skips or mis-keys), keys before any section header (git records the sectionless name `orphan`; tsgit drops it), and the `=`-path key grammar (`bad!key = v` is `fatal: bad config line` in git; tsgit accepts a garbage key). All one bucket: the fix is parsing entry content continuation after a header token instead of line classification. 24.9h pinned the no-`=` half of the grammar; this entry closes the `=` half + same-line constructs. _(surfaced by 24.9h.)_
- [ ] **24.9n** Flat-section `renameSection`/`removeSection` — `git config --rename-section a b` works on subsection-less sections (pinned: `[a]\n\tkey\n` → `[b]\n\tkey\n`); tsgit's `parseSectionName` requires the dotted `<section>.<subsection>` form, so flat renames/removals are refused with `INVALID_OPTION`. Accept the flat form + pin with interop. _(surfaced by 24.9h's interop slice.)_
- [ ] **24.9o** Interop-helper env hardening — `interop-helpers.ts`'s `runGit`/`tryRunGit` scrub `GIT_*` and set `GIT_CEILING_DIRECTORIES` but spawn git with the inherited `HOME` and no `GIT_CONFIG_NOSYSTEM`, so porcelain-flow twins (e.g. `git remote add`, rename-section flows) run against the developer's global config while the pinned design matrices are produced under `env -i` + isolated `HOME`; same trap class as the `merge.conflictStyle=diff3` interop flake. Harden the helpers (isolated `HOME` + `GIT_CONFIG_NOSYSTEM=1`) and sweep existing interop suites for tests that silently depended on global config. _(surfaced by 24.9i's test review; same shape as the conflict-style diff3 trap.)_
- [ ] **24.9p** Checkout cannot replace a symlink with a regular file — `apply-changeset.ts`'s `writeFileEntry` writes the regular payload through the node adapter's creation-mode containment, which refuses when a symlink occupies the path, so `checkout` between branches that change a path's kind (symlink → file) throws where git succeeds; the merge-side writers already solved this with an unlink-then-write (`writeWorkingTreeFile`'s `rmIfExists`). Fix `writeFileEntry` the same way + pin with a twin-checkout interop; then drop the manual `unlinkSync` workarounds baked into `distinct-types-with-base-interop.test.ts`'s `setupWithBase`/S8/P1/Q6 and P5 setups. _(surfaced by 24.9j's interop slices.)_
- [ ] **24.9q** Merge-command tracked-dirty conflict-path refusal (S13 gap) — git refuses a merge when a *tracked* working file with local changes would be overwritten by conflict materialisation (`Your local changes to the following files would be overwritten by merge`); tsgit's `merge` pre-flights only the distinct-types *untracked* rename-target case, while the apply consumers (`cherry-pick`/`revert`/`rebase`/`stash` via `applyMergeToWorktree.findWouldOverwrite`) already guard tracked-dirty paths. Add the tracked-dirty guard to `merge`'s conflict write path (conflict-wide, not distinct-types-specific) + pin refusal parity with interop. _(pre-existing, surfaced by 24.9j's design pinning, scenario S13 in `design/distinct-types-with-base.md`.)_

### Wave D — consumer-spike feedback (sgd GitAdapter spike, 2026-06-10)

Gaps hit by a real downstream consumer driving tsgit as its git adapter; in dependency order.

- [ ] **24.10** **BUG: object reads fail past ~64 KiB of compressed zlib** — both loose and packed paths (65,365 compressed bytes OK; 67,618 FAIL). Real-world triggers: any blob >64 KiB compressed (StaticResources), or a flat tree with >~2.5k entries (every big org's `classes/` dir). Repro: commit one 140 KB random-bytes file, `git gc`, `repo.primitives.readBlob(id)` → throws; loose-path repro: 13k files in one dir, read the tree. Likely a fixed 64 KiB buffer/chunk limit in the streaming inflate. Highest priority of the wave — blocks real repositories.
- [ ] **24.11** Public type re-exports — branded/structured types (`ObjectId`, `TreeDiff`, `PatchResult`, …) are not reachable from the package's node entry; consumers resort to `Awaited<ReturnType<Repository['revParse']>>` gymnastics. Audit the public surface and re-export every type the facade's signatures mention.
- [ ] **24.12** Recursive `diff`/`diffTrees` — the tree diff is single-level: a changed directory surfaces as one `modify` entry with two tree ids, forcing consumers to hand-roll recursive expansion (~100 lines in the spike's `walkDiff`). Add a `recursive: true` option; facade-level `detectRenames` is per-level today, so cross-directory moves can't pair until this lands (prerequisite for 24.13).
- [ ] **24.13** Similarity rename detection — rename pairing is exact-match only (R100 via identical blob ids); content-similarity renames (git's default ≥50%) won't pair. Needs 24.12's recursive diff for cross-directory candidates.
- [ ] **24.14** Whitespace diff options — no `--ignore-all-space` family; the spike emulated it by strip-and-compare over modified blob pairs (O(blob bytes) per file). Surface structured whitespace modes on the diff options.
- [ ] **24.15** Blob streaming — `readBlob` materialises the whole `Uint8Array`; no streaming / size-tiered escalation equivalent (matters for multi-MB StaticResource blobs). Sequenced after 24.10's inflate fix.
- [ ] **24.16** `gitGrep` pattern grammar — the pattern is treated as a JS `RegExp`; git's default is POSIX basic regex, with `-E`/`-F` variants. Decide the faithful grammar (or expose a mode enum) per ADR.
- [ ] **24.17** Diff faithfulness odds & ends from the spike — LFS pointer handling is wired but untested (add fixtures/interop); file↔symlink type changes are dropped from diff output (matches `--diff-filter=AMD` — decide whether `T` entries should surface); `log` could grow a `maxParents`/roots filter (`rev-list --max-parents=0`) for cheap root-commit lookup (today: walk + `parents.length === 0`).

- [ ] **26.2** Centralize current-branch / default-remote resolution — the `HEAD → short branch name → branch.<name>.remote ?? 'origin'` (and bare "current branch name from HEAD") logic is inlined across `pull`, `fetch`, `push`, `branch`, `status`, and now `submodule` (init/sync base-URL). Behavior-preserving; cross-cutting, so deferred out of the 24.1a submodule PR per the architecture-pass blast-radius bound. _(surfaced by 24.1a.)_
- [x] **26.2a** `merge` worktree+index materialisation (faithfulness fix; realises ADR-215's deferred migration). `mergeRun`'s **non-conflict** outcomes update only the commit/ref and leave the working tree + index stale at `ours` — diverging from canonical git, which checks the merge result out to **both** the index and the working tree. Confirmed against real `git` for **both** paths: (1) **fast-forward** — `git merge --ff` advances the index + worktree to the new tip (a checkout); tsgit's FF branch does a bare `updateRef`. (2) **clean true-merge** — `git merge --no-ff` writes the merged tree to index + worktree; tsgit's `commitCleanMerge` commits the correct tree but writes neither. Repro (theirs-only-add: base `f`, ours `+a`, theirs `+m`): the merge commit's **tree** is correct (`f+a+m`) but `m` is absent from the **index** (`f+a`) and from the **working tree** (`FILE_NOT_FOUND m`); affects every clean merge whose result differs from `ours` (theirs-only adds, theirs-side edits/deletes, combined content merges). Only the **conflict** path materialises today (`persistConflictState` → `writeConflictingWorkingTree` + `buildConflictIndexEntries`, sparse-aware per ADR-076). Root cause: ADR-215 extracted the shared `applyMergeToWorktree` primitive (clean → `materializeTree` onto worktree+index with git's "local changes would be overwritten" dirty guard; conflict → markers + stage-1/2/3) but **explicitly deferred migrating `merge.ts` onto it** ("out of scope … risks merge's mutation-tested invariants … tracked for a later cleanup") — every other 3-way consumer (cherry-pick / revert / rebase / stash) already routes through it; `merge` is the lone holdout. Fix: route `merge`'s clean path (and FF checkout) through worktree+index materialisation, preserving the conflict path's `MERGE_HEAD`/`ORIG_HEAD`/`MERGE_MSG` + sparse handling. Pin byte-for-byte with a `merge-interop` case (twin git/tsgit asserting **index + worktree + tree** parity after FF and clean true-merge — theirs-only-add, theirs-side edit, combined content merge — plus the dirty-worktree refusal). Supersedes ADR-215's deferral note. · ADR 293 · `design/merge-worktree-index-materialisation.md` _(surfaced 2026-06-08 while investigating a suspected theirs-only-add worktree gap.)_

## Phase 27 — Test base rework (sequenced after Phase 26)

Restructure every test tier around minimal-but-complete coverage instead
of breadth-of-cases. Each tier optimises for a different target.
Companion to (not replacement for) the mutation-pyramid harness shipped
in Phase 19. Starts AFTER Phase 26 lands so 27.4 has the perf data it
needs to define hot paths.

- [ ] **27.1 Unit — minimise while preserving GWT discipline.** Keep one
  `it('Then …')` per distinct behaviour. Minimise by collapsing tests
  that prove the SAME behaviour with different inputs into one
  parameterised test (`it.each`). Delete strict-subset tests outright.
  Outcome: 100 % line / branch / function / statement coverage AND
  only provably-equivalent mutants surviving (documented inline per
  existing convention). Intention-revealing test titles preserved.
- [ ] **27.2 Integration — collapse same-scenario-different-input
  overlap.** Two integration tests overlap when they exercise the same
  user journey or code path with different fixtures. Such tests merge
  into one parameterised case; strict-subset tests delete.
  **Parity cross-products (Node × Memory × Browser × Deno × Bun ×
  Workers) are NOT overlap** — they prove cross-adapter equivalence
  and stay distinct.
- [ ] **27.3 E2E — same overlap definition.** Each Playwright / browser
  flow asserts a user journey no other E2E flow asserts. Parity
  cross-products excluded from the overlap audit.
- [ ] **27.4 Perf — rebuild bench suite around hot paths.** Hot-path
  list is NOT pre-frozen here; it's derived from the Phase 26 perf
  pass output (the perf pass produces measurements that pick the
  hottest operations). Once that list lands as an ADR, hot paths get
  small / medium / large fixtures; non-hot paths keep medium only.
  Bench gating only on hot paths.
- [ ] **27.5 Mutation runner — Stryker × vitest-4 test/mutant mis-pairing.**
  Stryker 9.6.1's vitest-runner with vitest 4.1.8 (both latest) under-reports
  killed mutants: for some source files it runs a *subset* of the covering
  tests (false survivors) and for others finds *no* related tests at all
  (false `NoCoverage`). Surfaced during 23.6's mutation grind — e.g.
  `correspond.test.ts`'s pure-deletion test demonstrably kills the
  `if (j >= 0 && j < m)` → `true` mutant (proven by activating it via
  `__STRYKER_ACTIVE_MUTANT__` in the sandbox: baseline passes, mutant active
  fails), yet Stryker reports it surviving under **every** config permutation
  tried (`coverageAnalysis` perTest/all/off, `vitest.related` on/off, the
  `projects` config vs a dedicated flat config, `ignoreStatic` on/off,
  `mutantActivation` runtime/static, with/without `dir`, caches cleared) —
  *including* `off` mode, which is supposed to run all tests per mutant.
  `patch-text.test.ts` and the `range-diff` command test show the same as
  all-`NoCoverage`. Not config-fixable and no newer versions exist. Options to
  investigate: pin the Stryker vitest config to `pool: 'threads'`/`'vmThreads'`
  (vitest 4 defaults to `forks`, whose process isolation may starve Stryker's
  coverage/activation hooks), evaluate a vitest-3 line for the mutation job
  only, or report upstream. Until fixed, the mutation gate under-counts kills,
  so per-file scores must be read with the known false survivors/NoCoverage in
  mind.
- [ ] **27.6 `sut`/`result` naming sweep.** Legacy unit files (e.g.
  `apply-merge-to-worktree.test.ts`, `index-diff.test.ts`) bind the call
  result to `sut`; the convention is `sut` = the unit under test, with the
  outcome in `result`. Mechanical, file-by-file rename — new tests in those
  files currently follow the local (wrong) style for in-file consistency.
  _(surfaced by 24.9f's review pass.)_
